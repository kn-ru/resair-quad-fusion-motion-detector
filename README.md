# Quad Fusion Motion Detector

## Краткий реферат

Программа решает задачу **определения движущихся объектов (лодок) на видео с дрона** путём фузии четырёх независимых сигналов движения. В отличие от классических подходов, использующих только оптический поток или трекинг, данный метод объединяет:

1. **Детекцию кильватерного следа нейросетью** (YOLOv9, класс wake)
2. **Локально-компенсированный оптический поток** (Farneback с вычитанием фона вокруг объекта)
3. **Анализ траектории по IoU-трекингу** (смещение bbox между кадрами)
4. **Текстурный детектор водного следа** (белая пена за лодкой в HSV-пространстве)

Ключевая особенность — **устойчивость к слежению камеры за объектом**: когда дрон летит за лодкой, классические методы (optical flow, trajectory) дают нулевой сигнал, так как объект остаётся на месте в кадре. Сигналы 1 и 4 (wake-детекция и текстура пены) решают эту проблему, опираясь на физический признак движения — кильватерный след на воде.

Финальное решение принимается через **взвешенную фузию с гистерезисом**, что исключает мерцание статуса MOVING/STATIC между кадрами.

## Подробное описание каждого компонента

### 1. Детекция объектов — YOLOv9

Модель YOLOv9c обучена на 2 класса:
- **boat** (класс 0) — лодки, катера, суда
- **wake** (класс 1) — кильватерный след на воде

Обучение проводилось на датасете `boat_wake` (100 эпох, batch=16, imgsz=640, 2×GPU). Финальные метрики: mAP50=0.729, mAP50-95=0.453.

**Фильтрация ложных детекций:**
- Bbox, занимающие >10% площади кадра, отбрасываются (ложные срабатывания на крупных объектах)
- Bbox лодок, касающиеся края кадра (<5px от границы), отбрасываются (причалы, берег)
- Wake-детекции НЕ фильтруются по краю, т.к. кильватерный след часто выходит за границу кадра

### 2. Сигнал 1 — Wake Association (W)

**Идея:** Если нейросеть детектирует wake рядом с лодкой — лодка однозначно движется.

**Алгоритм:**
- Для каждой лодки проверяем все wake-детекции
- Если IoU(boat, wake) > 0 → score = 1.0
- Если расстояние центров < 150px → score экспоненциально убывает с расстоянием: `score = exp(-dist / 50)`
- Берём максимальный score среди всех wake

**Вес в фузии:** 0.30

### 3. Сигнал 2 — Optical Flow (F)

**Идея:** Движущийся объект создаёт оптический поток, отличный от фона.

**Алгоритм:**
1. Вычисляем dense optical flow (Farneback) между текущим и предыдущим кадром
2. Для каждого bbox лодки:
   - Берём кольцо шириной 40px ВОКРУГ bbox (фон)
   - Вычисляем медиану потока в кольце — это **движение камеры** в данной точке
   - Вычитаем фоновый поток из потока внутри bbox — получаем **собственное движение объекта**
3. 75-й перцентиль магнитуды компенсированного потока → нормализация в [0, 1]

**Локальная компенсация камеры** — ключевое отличие от глобальной медианы. Глобальная компенсация не работает, когда разные части кадра двигаются по-разному (поворот камеры, параллакс). Локальное кольцо вокруг bbox даёт точную оценку движения фона именно в месте нахождения объекта.

**Ограничение:** Когда дрон точно следит за лодкой, фон вокруг bbox двигается так же, как сама лодка → компенсированный поток ≈ 0. Поэтому этот сигнал дополняется сигналами 1, 3, 4.

**Вес в фузии:** 0.20

### 4. Сигнал 3 — Water Trail Texture (Tr)

**Идея:** Движущаяся лодка оставляет за собой белую пену и волны на воде. Эту текстуру можно детектировать по цвету, даже если камера следит за лодкой.

**Алгоритм:**
1. Берём расширенную зону вокруг bbox лодки (×2.5 от размера bbox)
2. Исключаем сам bbox (чтобы корпус лодки не влиял)
3. Переводим в HSV и ищем пиксели с:
   - Высокая яркость (V > 180) — белая пена
   - Низкая насыщенность (S < 80) — отсечение зелёной воды, деревьев
4. Считаем долю таких пикселей от общего количества в зоне поиска
5. Линейная нормализация: 2% ярких пикселей → score=0, 15% → score=1.0

**Почему это работает:** На спокойной воде практически нет белых/ярких пикселей (вода зелёная/синяя/тёмная). Кильватерный след — это физическое явление (турбулентность, пена, аэрация воды), которое создаёт яркие белые области за лодкой. Этот признак не зависит от движения камеры.

**Вес в фузии:** 0.35 (самый высокий — работает в любых условиях)

### 5. Сигнал 4 — Trajectory Displacement (T)

**Идея:** Если bbox лодки смещается между кадрами — лодка движется.

**Алгоритм:**
1. IoU-based трекинг: сопоставление bbox между кадрами по максимальному IoU (порог 0.25)
2. Хранение истории центров bbox (до 15 кадров)
3. Суммарное смещение центра за последние 5 кадров
4. Нормализация по диагонали кадра: смещение > 3% диагонали → score=1.0

**Ограничение:** Когда дрон следит за лодкой, bbox остаётся примерно на месте → displacement ≈ 0.

**Вес в фузии:** 0.15

### 6. Фузия и принятие решения

**Взвешенная сумма:**
```
raw_score = 0.30×W + 0.20×F + 0.15×T + 0.35×Tr
```

**Бусты (правила-исключения):**
- Если W > 0.7 (модель уверенно видит wake) → score ≥ 0.80
- Если Tr > 0.5 (много пены) → score ≥ 0.65
- Если W > 0.7 И Tr > 0.3 → score ≥ 0.90

**Гистерезис (анти-мерцание):**

Проблема: raw score колеблется между кадрами → статус мигает MOVING↔STATIC. Решение — двойной механизм стабилизации:

1. **EMA-сглаживание** (Exponential Moving Average, α=0.3):
   ```
   ema_score = 0.3 × raw_score + 0.7 × prev_ema_score
   ```
   Сглаживает кратковременные скачки.

2. **Гистерезис с двумя порогами:**
   - STATIC → MOVING: ema_score ≥ 0.30
   - MOVING → STATIC: ema_score < 0.15 **в течение 8 кадров подряд**

   Это означает, что объект, однажды определённый как движущийся, останется движущимся до тех пор, пока сигнал не будет стабильно низким в течение ~0.27 секунды (при 30fps).

## Визуализация

На каждом кадре отображается:
- **Bbox лодки** — красный (MOVING) или зелёный (STATIC)
- **Bbox wake** — оранжевый
- **Полоска motion score** — заполненная шкала над bbox
- **Текст** — `MOVING 0.85 [W:1.0 F:0.2 T:0.1 Tr:0.8]` — статус, EMA-score, и значение каждого из 4 сигналов
- **Стрелки оптического потока** — жёлтые стрелки внутри bbox (только для MOVING)
- **Траектория** — линия истории центра bbox
- **Инфо-панель** — общее количество лодок и движущихся

## Запуск

```bash
python motion_detector.py --video test.mp4 --weights train10/weights/best.pt
```

**Параметры:**

| Параметр | По умолчанию | Описание |
|----------|-------------|----------|
| `--video` | (обязательный) | Путь к входному видео |
| `--weights` | (обязательный) | Путь к весам YOLOv9 (.pt) |
| `--conf` | 0.3 | Порог уверенности детекции YOLO |
| `--imgsz` | 640 | Размер входного изображения YOLO |
| `--output` | `<video>_motion.mp4` | Путь к выходному видео |
| `--show` | false | Показывать в реальном времени (окно OpenCV) |

**Выход:** видеофайл с аннотациями + лог в консоль каждые 50 кадров.

## Конфигурация

Все параметры задаются константами в начале `motion_detector.py`:

| Параметр | Значение | Назначение |
|----------|----------|------------|
| `W_WAKE` | 0.30 | Вес сигнала wake-ассоциации |
| `W_FLOW` | 0.20 | Вес сигнала optical flow |
| `W_TRAJECTORY` | 0.15 | Вес сигнала trajectory |
| `W_TRAIL` | 0.35 | Вес сигнала текстуры водного следа |
| `MAX_BBOX_AREA_RATIO` | 0.10 | Макс. доля bbox от площади кадра |
| `EDGE_MARGIN_PX` | 5 | Отступ от края для фильтрации boat |
| `FLOW_LOCAL_MARGIN` | 40 | Ширина кольца фона для компенсации камеры (px) |
| `TRAIL_BRIGHT_THRESH` | 180 | Порог яркости белой пены (V в HSV) |
| `TRAIL_SEARCH_MULT` | 2.5 | Множитель зоны поиска пены вокруг bbox |
| `EMA_ALPHA` | 0.3 | Коэффициент EMA-сглаживания |
| `HYSTERESIS_ON` | 0.30 | Порог перехода STATIC → MOVING |
| `HYSTERESIS_OFF` | 0.15 | Порог перехода MOVING → STATIC |
| `MIN_STATIC_FRAMES` | 8 | Кадров подряд ниже порога для STATIC |
